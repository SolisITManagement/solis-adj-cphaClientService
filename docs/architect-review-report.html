Architect Questions

The Solis DIS (Drug Information System) platform is a cloud-native microservices architecture designed to facilitate HL7 messaging between Canadian healthcare providers and the BC PharmaNet system. It processes pharmacy transactions (dispensing claims, patient lookups, prescription management) by transforming simplified business JSON into standards-compliant HL7 v2 and v3 messages.

The system is deployed on Google Cloud Platform in the Montreal (northamerica-northeast1) region, using Cloud Run for compute, Firestore for persistence, and Pub/Sub for asynchronous inter-service messaging.

This document presents 39 questions a senior principal architect would ask during a compliance, cybersecurity, and regulatory review, along with suggested answers that reflect both the current state and the planned target state after the 27-day security hardening roadmap.

4. End-to-End Data Flow

1
User submits request — Careverse Angular UI (or test Client Service) collects business data (patient, prescription, pharmacy). Client Service’s BCPayloadBuilder wraps it into a full JSON envelope with Header, Sender, Receiver, MSH, Event, Candidate, Visit, and PharmaNet Z-segments. A unique messageId is generated.
2
Data Service receives and orchestrates — POST /api/dis/bc/publish persists the inbound request to Firestore collection dis-bc-inbound-requests (schema: PHARMANET_INBOUND_V1), publishes the full JSON to Pub/Sub topic dis-bc-request with attribute correlationMessageId, then begins polling Firestore dis-bc-responses (up to 225 retries at 200ms intervals).
3
Pub/Sub delivers via push — GCP Pub/Sub pushes the message as an HTTP POST to the Enrichment Service at /api/pharmanet/push-handler. The push envelope contains Base64-encoded message data and publishTime.
4
Enrichment Service processes — Decodes Base64, checks message age (drops stale >120s), extracts eventTypeCode and versionCode. Routes to the correct builder: HL7 v3 XML for V3PR1, HL7 v2 PharmaNet builders for pharmacy transactions, or ADT builders for admit/discharge events. Builds the HL7 message, sends to PharmaNet endpoint (real or SIMULATED), and writes the response to Firestore dis-bc-responses keyed by messageId.
5
Response returned — Data Service’s poll detects the new Firestore document. Returns the full response (including hl7Request, hl7Response, jsonRequest, timestamps) as HTTP 200 to the Client Service. The client renders three panels: HL7 Request, HL7 Response, and Full Database Record.

Category 1: Data Sovereignty & Jurisdictional Compliance

1
Where does PHI reside at rest, in transit, and during processing? Can you prove it never leaves Canadian borders?
Medium Risk
Suggested Answer: All PHI resides exclusively within Canadian borders. Our GCP infrastructure is deployed to northamerica-northeast1 (Montreal). Firestore stores all documents in the Montreal region. Cloud Pub/Sub messages are stored in the same region as the topic. Cloud Run containers process data in Montreal. Cloud Logging is configured to the same region.

To strengthen this answer: We will configure an Organization Policy constraint (gcp.resourceLocations) to enforce that all resources must be created in Canadian regions only. We will also ensure Cloud Logging sinks route exclusively to Canadian-region Cloud Storage or BigQuery datasets. A formal data residency audit will be conducted to verify that no GCP service (including Artifact Registry build logs, error reporting, or support diagnostics) replicates data outside Canada.
2
Which provincial health information acts govern this data?
High Risk
Suggested Answer: The following legislation applies:
  • BC Freedom of Information and Protection of Privacy Act (FIPPA) — Governs how public bodies collect, use, and disclose personal information
  • BC E-Health (Personal Health Information Access and Protection of Privacy) Act — Specifically governs the PharmaNet database and access to it
  • BC Pharmaceutical Services Act and PharmaCare Policies — Regulate pharmacy transactions and PharmaNet software certification
  • PIPEDA (Personal Information Protection and Electronic Documents Act) — Federal privacy legislation applicable to commercial health data processing
  • BC Personal Information Protection Act (PIPA) — Provincial private-sector privacy law
Our architecture is designed so that technical controls (access control, audit logging, encryption, data residency) map to the requirements of each act. A formal legal-to-technical controls mapping document will be produced as part of the compliance certification phase.
3
Is there a Data Protection Impact Assessment (DPIA)?
Critical Gap
Current State: No DPIA has been conducted.
Suggested Answer: A Data Protection Impact Assessment is planned and will be completed prior to production deployment. The DPIA will assess: (1) the nature and scope of PHI processing, (2) data flows across services and GCP components, (3) risks to patient privacy, (4) safeguards implemented (encryption, access control, audit logging, data minimization), and (5) residual risk assessment. The DPIA will be submitted to the BC Office of the Information and Privacy Commissioner (OIPC) as part of our Privacy Impact Assessment (PIA) process, which is mandatory under BC FIPPA for new systems processing personal health information.
4
Who is the data custodian vs. data processor? Is that contractually defined?
High Risk
Suggested Answer: The data roles are as follows:
  • Data Custodian: The pharmacy or health authority submitting transactions through Careverse. They determine the purpose and means of PHI processing.
  • Data Processor: Solis / Careverse, operating the platform that processes PHI on behalf of the custodian.
  • Sub-Processor: Google Cloud Platform, providing infrastructure services (compute, storage, messaging).
A Data Processing Agreement (DPA) will be executed with Google Cloud (Google provides a standard DPA for healthcare workloads that covers PIPEDA requirements). A Business Associate Agreement (BAA) equivalent will be established with each pharmacy or health authority client. These agreements will define data handling obligations, breach notification requirements, data retention, and sub-processor governance. Estimated completion: 2 days within the security hardening roadmap.
5
Does your Firestore TTL (30-day expireOn) satisfy both minimum retention and right-to-deletion requirements?
Medium Risk
Suggested Answer: The current 30-day TTL on Firestore response documents is designed for operational data minimization — we retain only what is needed for active transaction correlation and troubleshooting. However, this must be reconciled with:
  • BC Pharmaceutical Services Act: Pharmacy records may require longer retention (typically 3+ years for prescription records)
  • Audit requirements: Audit trails should be retained for the period required by the applicable compliance framework (e.g., 7 years for SOC 2)
Our approach: Operational Firestore documents will maintain the 30-day TTL for performance. Before TTL expiry, a scheduled Cloud Function will export audit-relevant fields (messageId, eventTypeCode, timestamps — without PHI) to a long-term Cloud Storage archive with configurable retention. This separates the operational concern from the regulatory retention concern and follows the data minimization principle.

Category 2: Authentication, Authorization & Identity

6
Show me the authentication chain end-to-end. Who authenticates the user, and how is that identity propagated to each service?
Critical Gap
Current State: No authentication exists at any layer. All endpoints are publicly accessible.
Suggested Answer (Target State): The authentication chain will operate as follows:
  • User → Careverse Angular: Users authenticate via the organization’s identity provider (e.g., Azure AD, Okta) using OpenID Connect. The Angular app receives a JWT access token.
  • Careverse → Data Service: The JWT is passed as a Bearer token in the Authorization header. Spring Security on the Data Service validates the JWT signature, issuer, audience, and expiry. User identity (roles, pharmacy ID) is extracted from claims.
  • Data Service → Pub/Sub: Service-to-service authentication via GCP IAM. The Data Service’s Cloud Run service account has pubsub.publisher role.
  • Pub/Sub → Enrichment Service: Google Pub/Sub attaches an OIDC token to push requests. The Enrichment Service validates the token using Google’s public keys, verifying the service account identity and audience.
  • Enrichment Service → PharmaNet: Authenticated via PharmaNet-specific credentials (pharmacy ID, software ID) embedded in HL7 MSH segment.
Implementation: Phase 1 of the security hardening plan (3 days).
7
How do you prevent a rogue actor from calling /api/dis/bc/publish directly with fabricated PHI?
Critical Gap
Current State: Anyone who discovers the Cloud Run URL can submit messages. No authentication, rate limiting, or input validation beyond Header.messageId presence.
Suggested Answer (Target State): Multiple layers of defense will be implemented:
  1. Cloud Run IAM: --no-allow-unauthenticated flag. Only authorized service accounts and authenticated users can invoke the service.
  2. JWT Validation: Spring Security filter chain validates Bearer tokens before any business logic executes.
  3. Cloud Armor WAF: Rate limiting, geo-restriction (Canada only), OWASP top-10 rules applied at the load balancer.
  4. Input Validation: JSON schema validation for all incoming payloads. Required fields, type checking, max lengths, and allowlisted eventTypeCode values.
  5. Audit Logging: Every request is logged with caller identity, IP, and timestamp for forensic analysis.
8
What is your service-to-service authentication model?
Critical Gap
Current State: Plain HTTP POST between Client Service and Data Service. No service identity, no mTLS, no API key.
Suggested Answer (Target State): Service-to-service authentication will use GCP IAM identity tokens:
  • Each Cloud Run service has a dedicated GCP service account with least-privilege IAM roles.
  • The Client Service (or Careverse backend) will obtain an identity token from the GCP metadata server and include it as a Bearer token when calling the Data Service.
  • Cloud Run’s built-in IAM authentication verifies the identity token before routing the request to the container.
  • For the Pub/Sub → Enrichment path, Google’s push subscription automatically attaches OIDC tokens signed by Google.
This model eliminates the need for application-level API keys or shared secrets between services.
9
How do you ensure the Pub/Sub push messages are actually from Google and not spoofed?
Critical Gap
Current State: The Enrichment Service’s push handler at /api/pharmanet/push-handler accepts any HTTP POST. No OIDC token verification. A spoofed request could inject fake PharmaNet responses into Firestore.
Suggested Answer (Target State): The Enrichment Service will verify Pub/Sub push authenticity using two mechanisms:
  1. Cloud Run IAM ingress: Setting --ingress=internal ensures only traffic from within the GCP project (including Pub/Sub) can reach the service. External HTTP requests are rejected at the infrastructure level.
  2. OIDC Token Verification: The push subscription will be configured with a service account. Google attaches a signed OIDC JWT to every push request. The Enrichment Service will validate this token using Google’s public JWKS endpoint, checking the iss, aud, and email claims.
Implementation: Phase 1.2 of the security hardening plan (1 day).
10
What is your session management and token lifecycle for the Angular client?
High Risk
Suggested Answer: Careverse (Angular) will implement standard OIDC session management:
  • Token acquisition: Authorization Code flow with PKCE (no implicit grant) via the organization’s IdP
  • Access tokens: Short-lived (15–30 minute expiry), stored in memory only (not localStorage)
  • Refresh tokens: Stored in HTTP-only, secure, SameSite cookies with rotation on each use
  • Session timeout: Configurable idle timeout (e.g., 15 minutes) with automatic token refresh
  • Logout: Token revocation at the IdP + clear local state
The specific IdP integration (Azure AD, Okta, or Keycloak) will be determined during the Careverse Angular authentication implementation, which is a separate workstream from the backend hardening.

12
What happens if someone submits a massive JSON payload to /api/dis/bc/publish?
High Risk
Suggested Answer: Defense-in-depth measures will be implemented:
  • Cloud Run: Default 32MB request body limit (configurable). This provides a baseline ceiling.
  • Spring Boot: server.max-http-request-size will be set to a reasonable limit (e.g., 256KB for PharmaNet payloads).
  • Application Layer: JSON schema validation on the Data Service will reject payloads that do not conform to the expected structure, including field-level max-length constraints.
  • Cloud Armor: Request size limits and rate limiting at the WAF layer to prevent volumetric abuse.
  • Firestore: Document size limit is 1MB (inherent protection against unbounded writes).
13
Your BCPayloadBuilder has a hardcoded MSH.security string in source code. What is that?
Critical Gap
Current State: The MSH.security field in BCPayloadBuilder.java contains a literal string that appears to be a credential or security token, committed to source control.
Suggested Answer: This will be remediated immediately:
  1. The hardcoded value will be removed from source code and replaced with a reference to Google Secret Manager.
  2. The secret will be stored in Secret Manager with appropriate IAM access controls (only the Data Service’s service account can read it).
  3. The git history will be audited. If this is a production credential, it must be rotated immediately after the migration.
  4. A pre-commit hook will be added to scan for potential secrets (e.g., using gitleaks or detect-secrets).
14
What prevents a Pub/Sub message replay attack?
Medium Risk
Suggested Answer: Multiple controls mitigate replay risk:
  • Message age check: The Enrichment Service rejects messages older than 120 seconds (based on publishTime), returning HTTP 204 to ACK without processing.
  • Idempotent writes: Firestore document ID = messageId. Duplicate processing overwrites the same document with identical data rather than creating duplicates.
  • Unique message IDs: Each request generates a cryptographically random messageId using SecureRandom, making collisions practically impossible.
Residual risk: PharmaNet calls are NOT idempotent — a replayed dispense claim could be processed twice at PharmaNet. Mitigation: the 120-second staleness window limits the replay window, and PharmaNet’s own duplicate detection (based on pharmacy ID + timestamp + Rx number) provides a secondary check.

16
Is PHI encrypted at the application level, or solely via GCP encryption at rest?
Medium Risk
Suggested Answer: Currently, we rely on GCP’s default encryption:
  • At rest: Firestore encrypts all data using AES-256 with Google-managed keys (default). Data is encrypted at the storage layer transparently.
  • In transit: All Cloud Run traffic is TLS 1.2+ (enforced by GCP). Pub/Sub messages are encrypted in transit.
Planned enhancement: For the most sensitive PHI fields (PHN, patient name, DOB), we will evaluate application-level field encryption using Google Cloud KMS before writing to Firestore, so that even a Firestore admin cannot read raw PHI without the application’s KMS key. We will also evaluate Customer-Managed Encryption Keys (CMEK) for Firestore to give us key rotation and revocation control.
17
Show me a log sample. Is PHI being logged?
Critical Gap
Current State: All JSON payloads (containing patient names, PHNs, prescriptions) flow through standard Java logging. PHI almost certainly appears in Cloud Run stdout captured by Cloud Logging.
Suggested Answer (Target State): PHI log redaction will be implemented via:
  1. logback-spring.xml: Custom PatternLayout with a regex-based converter that masks PHI patterns (10-digit PHN format, patient name fields, DOB patterns) before writing to stdout.
  2. Structured JSON logging: Moving from plain-text logs to structured JSON format using Logback’s LogstashEncoder — this enables Cloud Logging to parse fields and apply log-level filtering.
  3. Code-level discipline: Logger statements will be audited to ensure no logger.info(payload) or logger.debug(jsonRequest) statements exist that dump full PHI.
  4. Cloud Logging: A log sink will route to a restricted BigQuery dataset with IAM controls limiting who can query logs containing any residual PHI.
Implementation: Phase 2.5 of the security hardening plan (1 day).
18
What is your data minimization strategy?
Medium Risk
Suggested Answer: Data minimization is applied at multiple levels:
  • Client layer: Only business-relevant fields (patient name, PHN, DOB, prescription data) are collected from the user. The full HL7 envelope is assembled server-side — users never see or manipulate HL7 segments directly.
  • Firestore inbound: The dis-bc-inbound-requests collection stores the full payload for audit and correlation purposes. TTL-based expiry ensures documents are deleted after 30 days.
  • Firestore responses: Similarly TTL-controlled. Only the data needed for response correlation is stored.
  • Long-term archives: When implemented, only non-PHI audit fields (messageId, eventTypeCode, timestamps, success/failure) will be exported to long-term storage. PHI will NOT be archived beyond the operational TTL.
19
How do you handle patient consent for PHI processing through your cloud platform?
High Risk
Suggested Answer: Patient consent for PharmaNet transactions is governed by BC legislation:
  • Under the BC Pharmaceutical Services Act, pharmacists have statutory authority to access PharmaNet for the purpose of dispensing medications and providing pharmaceutical care. Explicit patient consent is not required for each transaction.
  • The PharmaNet Professional and Software Compliance Standards require that access is limited to authorized health professionals acting within their scope of practice.
  • Our platform enforces this through the authentication chain — only authenticated pharmacists/prescribers with valid PharmaNet credentials (embedded in HL7 MSH segments) can submit transactions.
  • Patient consent for broader data processing (e.g., analytics, quality improvement) would require separate consent mechanisms, which are outside the scope of the core PharmaNet messaging platform.

Category 5: Resilience, Availability & Operational Readiness

20
What is your SLA commitment and how does the architecture support it?
Medium Risk
Suggested Answer: Our target SLA is 99.9% availability (8.76 hours downtime/year). The architecture supports this through:
  • Cloud Run: Google provides a 99.95% monthly SLA for Cloud Run. Auto-scaling handles traffic spikes. Minimum instances can be configured to eliminate cold starts.
  • Firestore: 99.999% SLA for multi-region, 99.99% for single-region. Currently single-region; multi-region planned.
  • Pub/Sub: 99.95% SLA. Push delivery with automatic retries.
Current gap: Single-region deployment creates a single point of failure. The multi-region HA enhancement (Phase 3.3) will add Toronto as a DR region with automatic failover via Global HTTPS Load Balancer, achieving the target SLA.

Transaction timeout: The polling model provides ~45 seconds for a transaction to complete. If the timeout is reached, the client receives HTTP 504 and can retry using the /retry endpoint. The inbound request is already persisted and the Pub/Sub message is durable, so the transaction may still complete asynchronously.
21
What is the blast radius if Firestore goes down?
High Risk
Suggested Answer: Firestore is a critical dependency for both services:
  • Data Service: Cannot persist inbound requests or poll for responses. All new transactions fail with 500/504.
  • Enrichment Service: Can still receive Pub/Sub messages, build HL7, and send to PharmaNet, but cannot persist responses. This means the Data Service polling will never succeed.
Mitigations (planned):
  • Circuit breaker: Implement Resilience4j circuit breaker around Firestore calls. When open, return a graceful error with retry guidance.
  • Pub/Sub retry: If the Enrichment Service cannot write to Firestore, it returns a non-2xx status. Pub/Sub retries the push with exponential backoff, giving Firestore time to recover.
  • Multi-region Firestore: Moving to multi-region replication eliminates single-region Firestore outage as a risk.
  • Health checks: Spring Boot Actuator health endpoint will include a Firestore connectivity check, enabling Cloud Run to route traffic away from unhealthy instances.
22
What is your dead letter queue strategy for Pub/Sub?
High Risk
Current State: No dead letter queue configured. Failed messages are retried indefinitely by Pub/Sub until message retention expires.
Suggested Answer (Target State): A dead letter topic (dis-bc-request-dlq) will be configured on the dis-bc-request-sub subscription:
  • Max delivery attempts: 5 retries before routing to DLQ
  • DLQ monitoring: Cloud Monitoring alert when DLQ message count > 0
  • DLQ consumer: A lightweight Cloud Function or manual process to inspect, fix, and republish failed messages
  • DLQ retention: 7 days, giving operations time to investigate
23
How do you handle poison messages?
High Risk
Suggested Answer: A poison message (malformed JSON that consistently crashes the Enrichment Service) is handled through:
  1. Input validation: The Enrichment Service validates JSON structure before processing. Malformed messages return HTTP 400 (bad request), which Pub/Sub treats as a permanent failure after max retries.
  2. Dead letter queue: After 5 failed delivery attempts, poison messages route to the DLQ for manual inspection.
  3. Exception handling: The push handler catches all exceptions and returns appropriate HTTP status codes. Unrecoverable errors (bad JSON, unknown eventTypeCode) return 2xx to ACK the message and prevent infinite retries, while logging the error for investigation.
  4. Alerting: Spike in 400/500 responses from the Enrichment Service triggers a Cloud Monitoring alert.
24
What are your capacity limits? Have you load tested?
Medium Risk
Current State: No load testing has been conducted.
Suggested Answer: Known capacity characteristics:
  • Cloud Run: Auto-scales to 100 instances (default). Each instance handles concurrent requests (configurable concurrency).
  • Firestore: 10,000 writes/second per database. 1 write/second per document (not a concern since each transaction uses a unique messageId).
  • Pub/Sub: Effectively unlimited throughput for our scale.
  • Bottleneck: The polling model creates up to 226 Firestore reads per transaction. At 100 concurrent transactions, that is 22,600 reads in a ~45-second window.
Planned: Load testing using Cloud Run load testing tools or Locust to validate throughput, identify the actual bottleneck, and set appropriate Cloud Run concurrency and min-instance settings. Target: handle peak pharmacy hours (estimated 50–100 concurrent transactions) with P95 latency under 10 seconds.

Category 7: Regulatory & Compliance Specifics

29
Show me your PharmaNet conformance certification.
Critical Gap
Suggested Answer: PharmaNet Software Conformance certification is a mandatory requirement for production access to the PharmaNet system. The certification process, administered by the BC Ministry of Health, requires:
  • Submission of a completed PharmaNet software conformance application
  • Demonstration that all mandatory PharmaNet transactions are correctly implemented (our platform supports all 12 mandatory + 12 optional transaction types)
  • Passing conformance tests for HL7 message formatting, segment construction, and response handling
  • Review of security controls and privacy safeguards
Current status: The platform is in development/testing phase using the SIMULATED PharmaNet endpoint. Conformance certification will be pursued once security hardening is complete. Category 8: Architecture Patterns & Design

35
Why synchronous polling instead of webhooks or real-time listeners?
Design Debt
Suggested Answer: The polling model was chosen for simplicity and reliability:
  • Why not webhooks: The client needs a synchronous request-response pattern (submit transaction, get response). Webhooks would require the client to maintain a listener endpoint, complicating the architecture.
  • Why not Firestore real-time listeners: This is a valid optimization. The Data Service could use Firestore’s onSnapshot listener instead of polling, reducing Firestore reads from ~226 per transaction to 1–2. This is a candidate for a future optimization sprint.
  • Current cost: ~226 reads per transaction at ~$0.06/100K reads = negligible for expected volumes. The operational simplicity of polling outweighs the marginal cost at current scale.
Planned improvement: Migrate to Firestore real-time listeners when transaction volumes increase or if Firestore read costs become material.
36
Why is the Client Service using WebFlux WebClient in blocking mode (.block())?
Design Debt
Suggested Answer: The Client Service uses spring-boot-starter-webflux solely for the WebClient HTTP client (which replaces the deprecated RestTemplate). The .block() call is intentional because:
  • The service uses Thymeleaf (a synchronous template engine) which requires a completed response to render views
  • The Client Service is a test/demo application, not a high-throughput production service
  • Thread blocking is acceptable at the expected concurrency level (<10 concurrent users)
For the production Careverse Angular app, the Angular HttpClient is inherently asynchronous, so the blocking concern does not apply to the real client path.
37
Why three separate repositories instead of a monorepo with shared libraries?
Design Debt
Suggested Answer: The three-repository approach was chosen to support:
  • Independent deployability: Each service has its own CI/CD lifecycle. Changes to the Enrichment Service (e.g., adding a new HL7 builder) can be deployed without touching the Data Service.
  • Team scalability: As the team grows, different developers can own different services.
  • Technology flexibility: Services can evolve their dependencies independently (as seen in the Spring Boot version difference).
Trade-off acknowledged: Common domain types (Header, Event, Candidate) are duplicated. A shared domain library (solis-dis-common) published to a private Maven repository (Artifact Registry) would reduce this duplication. This is a candidate for a future refactoring sprint but is low priority given the current team size.
38
The Data Service is on Spring Boot 3.5.13 and Client Service on 3.2.5. Why the version mismatch?
Medium Risk
Suggested Answer: The version difference is historical — the Client Service was created earlier and has not been updated to match. This creates:
  • CVE risk: Spring Boot 3.2.5 may have known vulnerabilities patched in 3.5.x
  • Testing complexity: Different Spring behavior across services
Remediation: All services will be aligned to the same Spring Boot version (latest stable 3.5.x) during the CI/CD pipeline setup (Phase 3.2). The CI pipeline will enforce version consistency checks.
39
Why is the Enrichment Service both a message processor AND a PharmaNet client? Should those be separate?
Design Debt
Suggested Answer: The Enrichment Service combines three responsibilities: (1) Pub/Sub message intake, (2) HL7 message construction, and (3) PharmaNet HTTP communication. This was a pragmatic choice:
  • Latency: Separating message building from sending would add an extra network hop, increasing transaction latency in a time-sensitive pharmacy workflow.
  • Simplicity: The current 65+ class count is manageable with clear package separation (hl7.pnet, hl7.mandatory, hl7.optional, hl7.v3, controller, service).
  • Deployment unit: A single Cloud Run service simplifies infrastructure and reduces cost.
When to split: If the service grows beyond ~100 classes, if different scaling profiles are needed for message building vs. PharmaNet communication, or if a different team needs to own the PharmaNet gateway layer independently. Currently not a priority.
