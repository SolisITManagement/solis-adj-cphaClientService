BC PharmaNet – End-to-End Message Lifecycle

Last updated: April 2026

1. Architecture Overview

The BC PharmaNet integration platform processes healthcare messages through a three-tier, event-driven architecture running on Google Cloud Platform (GCP). Each message flows through the following layers:

Layer Service Role Technology
Client System CPHA Client Application Web-based testing UI that submits healthcare messages and displays responses Spring Boot 3.2, Java 21, Thymeleaf, WebClient
Orchestration Layer Data Service Receives requests, persists to Firestore, publishes to Pub/Sub, and polls for responses Spring Boot 3.2, Java 21, Firestore, Pub/Sub Publisher
Message Processing Engine Enrichment Service Builds HL7 messages from JSON, sends to PharmaNet, saves responses to Firestore Spring Boot 3.2, Java 21, Firestore, Pub/Sub Push

3. Detailed Flow Description

Step Layer Action Technical Detail
1 Client System User submits a message from the web UI User enters Patient & Prescription business data in a textarea, clicks Submit. BCPharmaNetController merges business data with infrastructure defaults (Header, Sender, Receiver, MSH, ZCA, ZCB, ZCC, etc.) via BCPayloadBuilder.buildFullPayload() to produce a complete JSON payload. A unique messageId is generated per request.
2 Orchestration Layer Validate and persist inbound request BCPublishController receives the JSON at POST /api/dis/bc/publish. PharmanetIngestionService extracts Header.messageId and persists the full payload to Firestore collection dis-bc-inbound-requests with the messageId as the document ID.
3 Orchestration Layer Publish to Pub/Sub PublisherService publishes the JSON payload to Pub/Sub topic dis-bc-request with a correlationMessageId attribute set to Header.messageId.
4 GCP Pub/Sub Push delivery to Enrichment Service Pub/Sub subscription dis-bc-request-sub (push mode) delivers the message as an HTTP POST to the Enrichment Service endpoint /api/pharmanet/push-handler on Cloud Run.
5 Message Processing Engine Decode Pub/Sub push envelope PharmaNetMessageController extracts the Base64-encoded message.data field, decodes to UTF-8 JSON, and checks TTL against message.publishTime. Stale messages are acknowledged without processing.
6 Message Processing Engine Route to the correct HL7 builder PharmanetEventService reads Event.eventTypeCode and Header.versionCode from the JSON:
  • If versionCode = V3PR1 → routes to HL7V3MessageBuilder (Find Candidate, Get Demographics)
  • Otherwise checks pnetBuilderMap (PharmaNet v2 transactions: TRP, TIP, TAC_TDU, etc.) then adtBuilderMap (ADT events: A01, DISPENSE, etc.)
7 Message Processing Engine Build HL7 message The matched builder constructs the HL7 message (v2 pipe-delimited or v3 XML) from the JSON input. Each builder is specialized for its transaction type (e.g., AdjudicateDispenseClaimBuilder, PatientProfileRequestBuilder, AdmitPatientMessageBuilder).
8 Message Processing Engine Send to PharmaNet PharmaNetClientService sends the HL7 message via HTTP POST to the PharmaNet endpoint. Content type is application/hl7-v2 for v2 messages or application/xml for v3.
9 Message Processing Engine Save response to Firestore FirestoreService.saveResponse() writes a document to Firestore collection dis-bc-responses with messageId as the document ID. The document contains: messageId, eventTypeCode, hl7Request, hl7Response, jsonRequest, createdAt, and expireOn.
10 Orchestration Layer Poll Firestore for response BCPublishController polls Firestore collection dis-bc-responses using the messageId as the document ID. It retries at a configurable interval until the response document is found or the timeout is reached (504 Gateway Timeout).
11 Client System Display response in web UI The client receives the full response JSON (including the HL7 request built by services, the HL7 response from PharmaNet, and metadata). The UI displays: the HL7 Request, the HL7 Response, and the full database response document.

9. Error Handling & Resilience

Scenario Behavior
Message Processing Engine is down Pub/Sub retries push delivery with exponential backoff. Messages are retained for up to 7 days.
Stale message (TTL exceeded) Message Processing Engine acknowledges (HTTP 204) without processing if publishTime exceeds the configured TTL (default 120s).
Unsupported event type Message Processing Engine returns HTTP 500. Pub/Sub retries. Logged as IllegalArgumentException.
PharmaNet unreachable HTTP timeout from PharmaNetClientService. Response not written to Firestore. Orchestration Layer returns 504 to Client System.
Response polling timeout Orchestration Layer retries Firestore reads at a configurable interval. Returns HTTP 504 with "Timed-out.. messageId not identified" if max retries exhausted.
