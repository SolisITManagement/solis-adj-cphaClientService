CareVerse → Solis DIS Integration Guide

How the CareVerse Angular/TypeScript application calls the Solis Drug Information Service (DIS) gateway for BC PharmaNet and NL Pharmacy Network messages.
Who this is for Engineers on the CareVerse team integrating with the Solis DIS gateway. CareVerse sends a JSON object to our gateway; Solis takes care of building the provincial wire formats (HL7 v2.1 for BC, CeRx XML v3 for NL) and delivering them to the provincial systems.

1. Overview & architecture

CareVerse produces a JSON payload. It is posted to a single external HTTPS gateway fronted by a GCP global external Load Balancer and protected by Cloud Armor. Inside GCP the request is routed to the Solis dataservice, which persists it, publishes it to Pub/Sub, and asynchronously hands off to the enrichmentservice. The enrichment service constructs the provincial message (HL7 v2.1 for BC, CeRx XML for NL), calls the provincial endpoint (PharmaNet for BC, HIAL for NL), and writes the response back to Firestore. CareVerse retrieves the response via a retry endpoint keyed by messageId.

3.4 Request/response contract

  • Request Content-Type: application/json.
  • All requests MUST include the header x-api-key: <shared secret>. See §4.
  • CareVerse supplies a messageId (UUID recommended). If omitted, Solis extracts or generates one and returns it.
  • Response body:
    • BC: HL7 v2.1 text (returned as a string inside JSON).
    • NL: CeRx HL7 v3 XML (returned as a string inside JSON).
  • Sample BC input JSON: src/main/resources/sampleCPhAAccepted.json.
  • NL input shape: see NLPayloadBuilder.java for the JSON the enrichment service expects for COMT_IN300001CA.

4. Authentication

The only thing on the wire is a shared API key. It is enforced by Cloud Armor in front of the Load Balancer, before the request reaches Cloud Run.

priority 100  : ALLOW if request.headers['x-api-key'] == '<shared secret>'
priority MAX  : DENY 403 (default)

Send the header on every request:

x-api-key: 00000000-0000-0000-0000-000000000000
Never ship the key to a browser bundle If CareVerse is a pure SPA, load the key in a backend-for-frontend (BFF) and proxy the call. See §5.

5. Integration paths

Two topologies are supported. Solis strongly recommends Path A (BFF).

Path A — BFF proxy Recommended

Angular browser → CareVerse BFF (Node/TS, Java, .NET — anything) → Solis gateway.

  • API key lives in GCP Secret Manager, read by the BFF's service account only.
  • Browser never sees x-api-key.
  • No CORS changes required on the Solis side.
  • BFF can add CareVerse user auth (OIDC/session) in front of the call.
  • Easier rotation and revocation.

Path B — Direct from browser Avoid if possible

Angular browser → Solis gateway (no CareVerse backend).

  • Any key shipped to the browser is effectively public.
  • Requires Solis to enable HTTPS + DNS (browsers block HTTPS to IP literals and block mixed content).
  • Requires Solis to emit CORS headers for CareVerse origins.
  • Mitigation: short-lived tokens minted by a Cloud Function after CareVerse OIDC auth, not a static API key.
  • Use only if CareVerse truly has no server-side component.

5.1 Path A — BFF walkthrough

  1. Stand up a small BFF (Node/TS is fine, so is Express, Nest, etc.). Deploy it in a CareVerse-owned GCP project or any VPC that can egress to 34.98.75.115.
  2. Give the BFF a dedicated service account, e.g. careverse-bff@<careverse-project>.iam.gserviceaccount.com.
  3. Grant it roles/secretmanager.secretAccessor on the Solis-owned secret — see §6.
  4. At BFF startup, fetch the secret value, cache it in memory (refresh every N minutes or use event-driven reload).
  5. Expose a thin REST surface that mirrors the Solis endpoints you need. Examples: POST /bff/dis/nl/publish, GET /bff/dis/nl/response/:messageId.
  6. On each BFF call, forward the JSON body, attach x-api-key: <secret value> and Content-Type: application/json, call ${GATEWAY_BASE_URL}/api/dis/nl/publish, relay the response.
  7. Angular calls the BFF — same-origin, no API key in sight.
Node/TS BFF — load key from Secret Manager + forward to Solis 
// secret-manager.ts
import { SecretManagerServiceClient } from '@google-cloud/secret-manager';

const sm = new SecretManagerServiceClient();
let cached: { value: string; expiresAt: number } | null = null;

export async function getSolisApiKey(): Promise<string> {
  const now = Date.now();
  if (cached && cached.expiresAt > now) return cached.value;

  const name = process.env.SECRET_RESOURCE_NAME!;
  // e.g. projects/solis-public/secrets/solis-dis-gateway-api-key-careverse/versions/latest

  const [version] = await sm.accessSecretVersion({ name });
  const value = version.payload?.data?.toString() ?? '';
  if (!value) throw new Error('Empty Solis API key');

  cached = { value, expiresAt: now + 10 * 60 * 1000 }; // cache 10 min
  return value;
}

// solis-client.ts
import { getSolisApiKey } from './secret-manager';

const BASE = process.env.GATEWAY_BASE_URL!; // e.g. https://dis-gw.solis.example

export async function submitNlAddPatientNote(payload: unknown) {
  const key = await getSolisApiKey();
  const res = await fetch(`${BASE}/api/dis/nl/publish`, {
    method: 'POST',
    headers: {
      'content-type': 'application/json',
      'x-api-key': key,
    },
    body: JSON.stringify(payload),
  });
  const text = await res.text();
  if (!res.ok) {
    throw new Error(`Solis NL publish failed: ${res.status} ${text}`);
  }
  return text;
}

// express-bff.ts (minimal)
import express from 'express';
import { submitNlAddPatientNote } from './solis-client';

const app = express();
app.use(express.json({ limit: '1mb' }));

app.post('/bff/dis/nl/publish', async (req, res, next) => {
  try {
    const result = await submitNlAddPatientNote(req.body);
    res.type('application/json').send(result);
  } catch (e) { next(e); }
});

app.listen(8080);
Angular (standalone component-style) service calling the BFF 
// solis-dis.service.ts
import { Injectable, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

export interface NlAddPatientNoteRequest {
  messageId: string;
  sender: { providerId: string; softwareId: string };
  patient: { phn: string; firstName: string; lastName: string; dob: string };
  location: { pharmacyId: string };
  annotation: { text: string; enteredBy: string; enteredAt: string };
}

export interface SolisRawResponse {
  messageId: string;
  payload: string; // HL7v2.1 (BC) or CeRx XML (NL) body
}

@Injectable({ providedIn: 'root' })
export class SolisDisService {
  private http = inject(HttpClient);
  private base = '/bff/dis'; // same-origin to the CareVerse BFF

  submitNlPatientNote(req: NlAddPatientNoteRequest): Observable<SolisRawResponse> {
    return this.http.post<SolisRawResponse>(`${this.base}/nl/publish`, req);
  }

  fetchNlResponse(messageId: string): Observable<SolisRawResponse> {
    return this.http.get<SolisRawResponse>(`${this.base}/nl/response/${messageId}`);
  }
}

5.2 Path B — Direct-from-browser walkthrough

Only feasible after Solis performs the following one-time changes. If you need Path B, open a ticket with the Solis DIS team first.

  1. Solis attaches a managed TLS cert and DNS name to the LB. Base URL becomes https://dis-gw.solis.example.
  2. Solis enables CORS on the Cloud Run services for the explicit CareVerse origin(s), allowing headers content-type, x-api-key and methods GET, POST, OPTIONS.
  3. CareVerse obtains a short-lived token from a Solis-owned Cloud Function (OIDC-gated). The browser puts that token in x-api-key. Static keys in Angular are NOT accepted.
  4. Solis adds a Cloud Armor rule keyed on that short-lived token (or IP range) at a dedicated priority.
Compliance note PHI flowing directly from browser to provincial gateways through a shared secret is almost never acceptable. If you can stand up any backend — even a tiny Cloud Run container — do Path A.

6. Secret Manager setup

All commands run with gcloud against project solis-public. The Solis team will run the create/versions add commands; CareVerse provides the BFF service account name for the IAM bindings.

Option 1 — Share the existing shared secret (quickest)

gcloud secrets add-iam-policy-binding solis-dis-gateway-api-key \
  --project=solis-public \
  --member="serviceAccount:<careverse-bff-sa>@<careverse-project>.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"

Option 2 — Dedicated per-client secret Recommended

# 1. Create the secret shell
gcloud secrets create solis-dis-gateway-api-key-careverse \
  --project=solis-public \
  --replication-policy=automatic

# 2. Add a version with a freshly generated UUID
echo -n "$(uuidgen)" | gcloud secrets versions add solis-dis-gateway-api-key-careverse \
  --project=solis-public \
  --data-file=-

# 3. Grant CareVerse's BFF SA read access to THIS secret only
gcloud secrets add-iam-policy-binding solis-dis-gateway-api-key-careverse \
  --project=solis-public \
  --member="serviceAccount:<careverse-bff-sa>@<careverse-project>.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"

Cloud Armor rule (Solis-side, per client)

# Solis team runs this after Option 2 above; <NEW_KEY> is the value stored in the secret.
gcloud compute security-policies rules create 110 \
  --project=solis-public \
  --security-policy=solis-dis-security-policy \
  --description="Allow CareVerse API key" \
  --action=allow \
  --expression="request.headers['x-api-key'] == '<NEW_KEY>'"
Why per-client secrets are worth it Independent rotation, per-client audit in Cloud Audit Logs, and a single gcloud compute security-policies rules delete 110 to revoke CareVerse without impacting anyone else.

8. Local / dev setup

  1. Developer authenticates once: gcloud auth application-default login — using a Google identity that has secretAccessor on the shared (or per-client) secret.
  2. Set environment variables for the BFF process: 
    export GATEWAY_BASE_URL="http://34.98.75.115"   # today
# export GATEWAY_BASE_URL="https://dis-gw.solis.example"  # after TLS+DNS rollout

export SECRET_RESOURCE_NAME="projects/solis-public/secrets/solis-dis-gateway-api-key-careverse/versions/latest"
  3. Run the BFF locally on http://localhost:8080.
  4. Run Angular with a dev-server proxy pointing /bff/* to http://localhost:8080.
  5. Smoke test from the shell without the BFF (sanity only): 
    curl -sS -I "$GATEWAY_BASE_URL/api/dis/nl/publish" -H "x-api-key: $SOLIS_API_KEY" -X POST
# Expect 400 (empty body) or 200, NOT 403. A 403 means the key is wrong or missing.

9. Test & verify checklist

  1. Confirm the BFF can read the secret: log a masked prefix like "key loaded: c9e22f14****" at startup, same pattern Solis uses in WebClientConfig.
  2. POST a minimal NL payload (see §7.1) through the BFF.
  3. Expect HTTP 200 and a JSON body containing a messageId.
  4. Within a few seconds, GET /api/dis/response/nl/retry?messageId=... and confirm a CeRx XML body comes back.
  5. Repeat for BC (/api/dis/bc/publish) with a payload based on sampleCPhAAccepted.json.
  6. In Cloud Logging (Solis-side), confirm the request appears in both solis-dis-dataservice and solis-dis-enrichmentservice with matching messageId.
  7. Try intentionally sending a wrong key — you should receive a 403 from Cloud Armor with no body, which confirms the policy is working.

11. Operations

Rotation

  1. Solis adds a new version to solis-dis-gateway-api-key-careverse: gcloud secrets versions add ... --data-file=-.
  2. Solis updates the Cloud Armor rule to accept the new value (or adds a second allow rule during overlap).
  3. CareVerse BFF picks up the new version on next cache expiry (configurable; default 10 min).
  4. Solis disables/destroys the old version after CareVerse confirms rollout.

Revoke CareVerse

gcloud compute security-policies rules delete 110 \
  --project=solis-public \
  --security-policy=solis-dis-security-policy

That single command stops all CareVerse traffic at the edge, without affecting any other tenant.

Audit

  • LB access logs: resource.type="http_load_balancer" AND resource.labels.url_map_name="solis-dis-url-map" in Cloud Logging.
  • Secret Manager access: protoPayload.serviceName="secretmanager.googleapis.com" — tracks who read the key and when.
  • Per-client attribution: the Cloud Armor rule priority (110 = CareVerse) is present on every allowed request, so filter by jsonPayload.enforcedSecurityPolicy.outcome="ACCEPT" AND jsonPayload.enforcedSecurityPolicy.priority=110.

12. Glossary

TermMeaning
LBGCP global external HTTP(S) Load Balancer in front of Solis services.
URL mapLB component that routes paths to backend services. Ours routes /api/dis/* to the dataservice and other paths to the adjudicate backend.
Cloud ArmorGCP WAF/edge policy. Currently enforces an x-api-key header match and default-denies everything else.
BFFBackend-for-frontend. A small server that a single-page app (like Angular) talks to, which in turn calls upstream APIs. Used to keep secrets out of the browser.
Pub/Sub pushGCP messaging delivery mode where Pub/Sub HTTP-POSTs each message to a subscriber URL. Used to hand off from dataservice to enrichmentservice.
HL7 v2.1Pipe-and-caret text format used by BC PharmaNet.
CeRxCanadian e-prescribing HL7 v3 XML standard. Used by NL HIAL.
HIALHealth Information Access Layer — NL's provincial integration gateway.
Secret ManagerGCP service for storing and versioning secrets. Accessed with IAM; never put secrets in source.
