Architecture Diagram

The Solis DIS platform follows a three-layer architecture: Client System (Careverse Angular + test client), Orchestration Layer (Data Service), and Message Processing Engine (Enrichment Service). All services run on Google Cloud Platform in the Montreal region.

Q1: How Do You Approach API Authentication in Enterprise Architecture?

1
API Authentication in Enterprise Architecture
Critical Gap

What We Built

  • GCP Application Default Credentials — Both the Data Service (FirestoreConfig.java) and Enrichment Service use GoogleCredentials.getApplicationDefault() to authenticate with Firestore and Pub/Sub. This is infrastructure-level authentication between our code and GCP managed services.
  • Cloud Run IAM (partial) — The Enrichment Service’s cloudbuild.yaml deploys with --no-allow-unauthenticated, meaning GCP’s IAM layer requires a valid identity token to invoke it. This is platform-level auth, not application-level.
  • PharmaNet Credentials in HL7 — Every HL7 message built by the Enrichment Service includes MSH segment credentials (pharmacy ID, software ID, security token) required by PharmaNet. This authenticates our system to the external PharmaNet endpoint at the HL7 protocol level.
  • Pub/Sub Publisher Identity — The Data Service’s PublisherService.java authenticates to the Pub/Sub topic using the Cloud Run service account’s IAM role (pubsub.publisher). This is automatic and handled by the GCP client library.

Where the Gaps Are

  • No Spring Security — None of the three services include spring-boot-starter-security in their pom.xml. There is no SecurityFilterChain, no HTTP filter chain, no @PreAuthorize annotations.
  • No JWT / Bearer Token Validation — The Data Service’s BCPublishController accepts any HTTP POST to /api/dis/bc/publish without checking an Authorization header. There is no token introspection, no JWKS validation, no issuer/audience checks.
  • No OIDC Push Verification — The Enrichment Service’s PharmaNetMessageController.handlePubSubPush() accepts any HTTP POST to /api/pharmanet/push-handler. It does not verify the OIDC token that Google Pub/Sub attaches to push requests. A spoofed request could inject data.
  • No RBAC — There is no concept of user roles. A pharmacist, technician, and unauthorized user would all have the same access to every endpoint and every PharmaNet transaction type.
  • No Service-to-Service Auth — The Client Service calls the Data Service with a plain WebClient.post() in CphaClientService.java — no Bearer token, no API key, no mTLS.

How to Fill the Gap

  1. Add spring-boot-starter-oauth2-resource-server to the Data Service and Enrichment Service pom.xml files. Configure a SecurityFilterChain bean with .oauth2ResourceServer(jwt -> jwt.jwkSetUri(...)) pointing to the organization’s Identity Provider (Azure AD, Okta, or Keycloak).
  2. OIDC Push Verification — Add a servlet filter on the Enrichment Service that validates the Authorization: Bearer header on push requests against Google’s public JWKS endpoint (https://www.googleapis.com/oauth2/v3/certs). Verify iss, aud, and email claims match the configured Pub/Sub push service account.
  3. Cloud Run IAM — Add --no-allow-unauthenticated and --ingress=internal to ALL deploy.sh / cloudbuild.yaml scripts. This provides defense-in-depth at the GCP platform layer.
  4. RBAC — Extract user roles from JWT claims and enforce per-endpoint authorization. For example, only PHARMACIST role can submit TAC_TDU (dispense claim) transactions.
  5. Service-to-Service — Use GCP IAM identity tokens: the Client Service obtains an identity token from the metadata server and passes it as a Bearer token to the Data Service. Cloud Run verifies it automatically when --no-allow-unauthenticated is set.
Estimated: 3 days (Phase 1 of Security Hardening Plan)

Q2: What Role Does an API Gateway Play in a Secure Architecture?

2
API Gateway in Secure Architecture
Critical Gap

What We Built

  • Data Service as Implicit Gateway — The Data Service acts as the single entry point for all BC PharmaNet traffic. All client requests go through /api/dis/bc/publish. It validates the required Header.messageId, persists the audit trail, and publishes to the internal Pub/Sub topic. No client directly calls the Enrichment Service — only Pub/Sub does.
  • Cloud Run Built-in Routing — Google Cloud Run provides automatic HTTPS termination, request routing, and basic DDoS protection. Each service gets a stable HTTPS URL with a managed TLS certificate.
  • SpringDoc OpenAPI — Both the Data Service and Enrichment Service include springdoc-openapi-starter-webmvc-ui (v2.8.6), providing Swagger UI at /swagger-ui.html and OpenAPI 3 spec at /v3/api-docs. This gives API consumers a self-service discovery interface.

Where the Gaps Are

  • No Dedicated API Gateway — There is no Spring Cloud Gateway, Kong, Apigee, or any API gateway product. Cloud Run URLs are called directly by clients.
  • No Rate Limiting / Throttling — Any caller can send unlimited requests. There is no per-client quota, no burst limit, no concurrency cap at the edge.
  • No WAF / DDoS Protection at the Edge — No Cloud Armor WAF policies are attached. No OWASP top-10 rule set, no geo-restriction, no IP allowlisting.
  • No Centralized Auth Token Validation — Without a gateway, each service must independently validate tokens. This creates duplication and risk of inconsistency.
  • No API Versioning or Traffic Splitting — No ability to route v1 vs. v2 traffic, or do canary deployments at the gateway layer.

How to Fill the Gap

  1. Google Cloud Global HTTPS Load Balancer — Deploy a Global LB in front of both Cloud Run services. This provides a single stable IP/domain, SSL termination, and request routing to the correct backend service based on URL path.
  2. Cloud Armor WAF — Attach Cloud Armor security policies to the LB backend services:
    • OWASP ModSecurity Core Rule Set (CRS) for top-10 protection
    • Rate limiting: max 100 requests/minute per IP
    • Geo-restriction: allow only Canadian IP ranges
    • Custom rules: block requests with payload > 256KB
  3. Optional: Apigee API Management — If the platform needs API key management, developer portal, analytics, or per-client quotas, add Apigee as a full API gateway layer. This is a larger investment and may not be needed at current scale.
  4. API Versioning — Use URL path versioning (/api/v1/dis/...) with LB path-based routing to support rolling upgrades.
Estimated: 1 day (Phase 2.1 of Security Hardening Plan)

Q3: When Would You Use Asynchronous Communication Such as Kafka or Pub/Sub?

3
Asynchronous Communication with Pub/Sub
Strongest Area

What We Built

This is the most mature architectural pattern in the platform. We chose Google Cloud Pub/Sub over Apache Kafka for the following reasons and implemented it as follows:

  • Why Pub/Sub over Kafka: Serverless (no broker management), native GCP integration with Cloud Run and IAM, automatic scaling, pay-per-message pricing. Kafka would require provisioning and managing a cluster (or using Confluent Cloud), which is unnecessary for our message volumes.
  • Topic: dis-bc-request in GCP project solis-public
  • Subscription: dis-bc-request-sub configured for push delivery to the Enrichment Service at /api/pharmanet/push-handler
  • Publishing: PublisherService.java in the Data Service creates a PubsubMessage with UTF-8 JSON data and a correlationMessageId attribute (set to Header.messageId). This attribute enables end-to-end tracing.
  • Push Handler: PharmaNetMessageController.handlePubSubPush() in the Enrichment Service decodes the Base64 push envelope, extracts the JSON payload, and routes to the correct HL7 builder.
  • Stale Message Rejection: Messages older than 120 seconds (based on publishTime) are ACK’d with HTTP 204 without processing. This prevents stale backlog from overwhelming the system.
  • At-Least-Once + Idempotency: Pub/Sub guarantees at-least-once delivery. Idempotency is achieved through Firestore document ID = messageId. Duplicate deliveries overwrite the same document.
  • Decoupling: The Data Service and Enrichment Service have zero compile-time dependencies on each other. They communicate only through the Pub/Sub topic and Firestore collections. Either can be redeployed independently.

Where the Gaps Are

  • No Dead Letter Queue (DLQ) — If the Enrichment Service persistently fails to process a message (returns non-2xx), Pub/Sub retries indefinitely until the message’s retention period expires (default 7 days). There is no mechanism to capture and inspect failed messages.
  • No Poison Message Handling — A malformed JSON payload that crashes the Enrichment Service will be retried forever by Pub/Sub, consuming resources without resolution.
  • No Pub/Sub Monitoring Alerts — No Cloud Monitoring alert for subscription backlog growth, delivery failure rate, or push latency.
  • No OIDC Token Verification — The push handler does not verify that the incoming POST is genuinely from Google Pub/Sub (covered in Q1).

How to Fill the Gap

  1. Dead Letter Queue — Create topic dis-bc-request-dlq and configure the subscription with --dead-letter-topic and --max-delivery-attempts=5. After 5 failed deliveries, messages route to the DLQ for inspection.
  2. Poison Message Safety — In the Enrichment Service push handler, wrap all processing in a try/catch. For unrecoverable errors (invalid JSON, unknown eventTypeCode), return HTTP 200 to ACK the message (preventing infinite retry) and log the error at WARN level for investigation.
  3. Monitoring — Create Cloud Monitoring alerts:
    • Subscription unacked messages > 50 for 5 minutes
    • DLQ message count > 0
    • Push delivery latency P95 > 30 seconds
Estimated: 1 day (Phase 2.3 of Security Hardening Plan)

Q4: How Do You Approach Orchestration Patterns in Distributed Systems?

4
Orchestration Patterns in Distributed Systems
Partially Implemented

What We Built

  • Choreography Pattern via Pub/Sub — The Data Service publishes an event to dis-bc-request. The Enrichment Service reacts to it independently. Neither service knows the other exists — they communicate through the shared topic and Firestore collections. This is the classic choreography pattern where each service owns its behavior.
  • Lightweight In-Process Orchestration — The Data Service’s PharmanetPublishOrchestrator implements a sequential workflow:
    1. Load message bundle template (PharmanetMessageBundleLoader)
    2. Deep-copy and inject messageId (PharmanetPublishMessagePreparer)
    3. Persist to Firestore (FirestorePharmanetInboundPersistence)
    4. Publish to Pub/Sub (BcRequestPublisher)
    5. Poll for response (BcResponseFetcher)
    Each step is a separate class with a single responsibility, composed together in the orchestrator.
  • Correlation-Based Request/Response — The Header.messageId (generated by BCPayloadBuilder.generateRandomId() using SecureRandom) flows through the entire chain: Client → Data Service → Pub/Sub attribute → Enrichment Service → Firestore document ID. The Data Service polls Firestore by this ID to correlate the asynchronous response.
  • Builder Routing via Map Lookup — The Enrichment Service’s PharmanetEventService uses three @Qualifier-injected maps (mandatoryPnetBuilderMap, optionalPnetBuilderMap, adtBuilderMap) to route each eventTypeCode to the correct HL7 builder. This is a strategy/router pattern that decouples message intake from message construction.

Where the Gaps Are

  • No Saga / Compensation Pattern — If step 4 (Pub/Sub publish) succeeds but step 3 (Firestore persist) failed, there is no rollback or compensation logic. The message would be on Pub/Sub but the inbound audit trail would be missing. Similarly, if the Enrichment Service processes a message but fails to write the response to Firestore, the Data Service polls forever and returns 504.
  • No Circuit Breaker — There is no Resilience4j, Hystrix, or Spring Retry around Firestore calls or PharmaNet HTTP calls. A Firestore outage would cause cascading failures across all in-flight transactions without any graceful degradation.
  • Polling-Based Correlation — The Data Service polls Firestore 225 times (200ms intervals, ~45 seconds total) instead of using an event-driven notification. This is simple but resource-intensive.

How to Fill the Gap

  1. Add Resilience4j Circuit Breaker — Wrap Firestore and PharmaNet calls in circuit breakers. When the circuit opens (e.g., 50% failure rate over 10 calls), immediately return a meaningful error instead of attempting the call. Configure fallback behavior (e.g., queue the request for retry).
  2. Compensation Logic — In the Data Service orchestrator, if Pub/Sub publish fails after Firestore write, update the inbound document status to PUBLISH_FAILED. A scheduled cleanup process can re-attempt publishing for failed documents. In the Enrichment Service, if Firestore write fails after PharmaNet call, return non-2xx to Pub/Sub so it retries delivery.
  3. Future: Firestore Real-Time Listeners — Replace the polling loop with Firestore onSnapshot() listener. This would reduce Firestore reads from ~226 to ~2 per transaction and provide instant notification when the response arrives.
Estimated: 2 days (new work beyond current security plan)

Q5: How Do You Handle Caching in Enterprise Applications?

5
Caching in Enterprise Applications
Minimal / By Design

What We Built

  • HL7 Builder Map Initialization (Startup Cache) — The Enrichment Service’s PharmanetEventService.initBuilderMap() is called once at startup via @Autowired. It merges 3 configuration maps (mandatory, optional, ADT) into in-memory HashMap instances that persist for the lifetime of the application. This is effectively an immutable cache of builder routing logic — no repeated lookups or re-initialization.
  • Spring @Value Injection (Configuration Cache) — All configuration properties (Firestore collection names, Pub/Sub topic IDs, PharmaNet endpoint URLs, timeouts) are injected via @Value at bean creation time. Spring caches these for the bean’s lifecycle. There is no per-request property file reading.
  • Design Decision: No Response Caching — PharmaNet transactions are real-time, state-changing operations (e.g., dispensing a prescription, recording a claim). Each transaction is unique and must reach PharmaNet for processing. Caching responses would be incorrect and potentially dangerous in a healthcare context — a cached “claim accepted” response for a different patient could cause medication errors.

Where the Gaps Are

  • Firestore Polling Reads (Cost Concern) — The Data Service polls dis-bc-responses up to 225 times per transaction. On each poll, it reads the same document (which doesn’t exist yet). Once the document appears, it’s read once more. The 224 “not found” reads are wasted. A simple cache of the “not found” result per messageId wouldn’t help (we need to check for new data), but a Firestore real-time listener would eliminate this entirely.
  • No Reference Data Caching — If the Careverse Angular app needs drug reference lists, pharmacy location directories, or prescriber lookup data, there is no caching layer for this read-heavy, infrequently-changing data.

How to Fill the Gap

  1. Replace Polling with Firestore Listener — Use Firestore’s document.addSnapshotListener() in the Data Service. This eliminates all polling reads and provides near-instant response notification. Reduces Firestore reads from ~226 to ~2 per transaction.
  2. Reference Data Cache (if needed) — Add spring-boot-starter-cache with Caffeine as the in-memory provider. Apply @Cacheable to reference data lookups with a 1-hour TTL. For larger datasets or cross-instance consistency, use Google Cloud Memorystore (Redis).
  3. HTTP Response Caching Headers — For read-only reference endpoints (not PharmaNet transactions), add Cache-Control headers to allow client-side and CDN caching.
Estimated: 0.5 day for Firestore listener migration; caching layer if needed separately

Q6: How Do You Approach Capacity Management?

6
Capacity Management
Partially Implemented

What We Built

  • Cloud Run Auto-Scaling — Both services run on Cloud Run, which automatically scales instances from 0 to max based on incoming request rate. No manual capacity provisioning is needed for compute.
  • Enrichment Service Scaling Bounds — The cloudbuild.yaml specifies --min-instances=1 (eliminates cold starts) and --max-instances=5 (cost cap). Memory and CPU are configured per instance.
  • Firestore Auto-Scaling — Firestore is serverless and auto-scales to handle any throughput we can generate. No capacity provisioning required.
  • Pub/Sub Auto-Scaling — Pub/Sub is fully managed and handles any message throughput. Push delivery automatically adjusts to the consumer’s capacity.
  • Timeout-Based Backpressure — The Data Service’s 45-second polling timeout and the Enrichment Service’s 120-second message staleness check provide implicit backpressure. Transactions that can’t complete in time fail gracefully with HTTP 504.

Where the Gaps Are

  • Data Service has no scaling configuration — The deploy.sh script has no --min-instances, --max-instances, or --concurrency flags. Cloud Run defaults apply (0 min, 100 max, 80 concurrency), which may not be optimal.
  • No Load Testing — No load test has been performed. The theoretical bottleneck (Firestore polling at scale) has not been validated empirically.
  • No Capacity Monitoring Dashboard — No Cloud Monitoring dashboard tracks Cloud Run instance count, request latency, Firestore operations/second, or Pub/Sub backlog.
  • No Concurrency Tuning — The default Cloud Run concurrency of 80 requests per instance may be too high for the Data Service, where each request holds a thread for ~45 seconds during polling.

How to Fill the Gap

  1. Data Service Scaling Config — Add --min-instances=1 --max-instances=10 --concurrency=20 to deploy.sh. The low concurrency (20 vs. default 80) accounts for the long-polling behavior — each request occupies a thread for up to 45 seconds.
  2. Load Testing — Run a load test with Locust or k6: simulate 100 concurrent PharmaNet transactions, measure P50/P95/P99 latency, Firestore read throughput, and Cloud Run scaling behavior. Document the results as a baseline.
  3. Capacity Dashboard — Create Cloud Monitoring dashboard with:
    • Cloud Run: instance count, request count, request latency, CPU/memory utilization
    • Firestore: read/write ops per second, latency
    • Pub/Sub: unacked messages, push delivery latency
  4. Alerting — Alert when Cloud Run max instances > 80% capacity or Pub/Sub backlog exceeds 100 messages.
Estimated: 1 day (Phase 2.3 of Security Hardening Plan)

Q7: What Is the Difference Between Vertical and Horizontal Scaling?

7
Vertical vs. Horizontal Scaling
Partially Implemented

What We Built

Our architecture inherently supports both scaling dimensions through Cloud Run:

  • Horizontal Scaling (adding more instances) — Cloud Run auto-scales from min-instances to max-instances based on incoming traffic. Each instance is a stateless container. The Enrichment Service scales from 1 to 5 instances (per cloudbuild.yaml). This is the primary scaling mechanism.
  • Vertical Scaling (bigger instances) — The Enrichment Service configures --memory and --cpu per instance in cloudbuild.yaml. This controls how much resource each individual container gets.
  • Stateless Design Enables Horizontal Scaling — All three services are stateless — no in-memory sessions, no local file state, no sticky sessions. State lives in Firestore (durable) and Pub/Sub (transient). Any instance can handle any request, enabling seamless horizontal scaling.
  • Serverless Data Layer — Firestore and Pub/Sub scale independently of the application tier. There is no manual sharding, partitioning, or replica management.

Where the Gaps Are

  • No Documented Scaling Strategy — There is no written policy on when to scale vertically (increase memory/CPU per instance) vs. horizontally (increase max instances). Engineering decisions are ad hoc.
  • Concurrency Not Tuned — Cloud Run’s --concurrency setting determines how many concurrent requests a single instance handles. This directly affects whether we need horizontal scale (more instances) or can handle more with existing ones (higher concurrency). Currently untuned.
  • No Auto-Scaling Policies Beyond Cloud Run Defaults — Cloud Run scales based on request concurrency. There is no CPU-based or custom-metric-based scaling.

How to Fill the Gap

  1. Document Scaling Decision Framework:
    • Scale horizontally (more instances) when throughput exceeds what a single instance can handle at acceptable latency. This is the default and preferred approach for Cloud Run.
    • Scale vertically (more memory/CPU) when a single request needs more resources — e.g., if HL7 v3 JAXB marshalling of large payloads causes OOM, increase --memory. Or if CPU-bound HL7 building becomes a bottleneck within a single request, increase --cpu.
  2. Tune Concurrency — Based on load test results (Q6), set --concurrency for each service. Data Service needs low concurrency (long-polling), Enrichment Service can handle higher concurrency (quick per-request processing).
Estimated: Included in load testing effort (Q6)

Q8: How Do You Approach Cost Efficiency as an Architectural Concern?

8
Cost Efficiency as Architectural Concern
Partially Implemented

What We Built

  • Serverless = Pay Per Use — The entire architecture is serverless:
    • Cloud Run: Billed per request and per vCPU-second/GiB-second of container time. Zero traffic = near-zero cost (when min-instances=0).
    • Firestore: Billed per document read/write/delete. No provisioned throughput.
    • Pub/Sub: Billed per message published/delivered. Negligible cost at current volumes.
  • No Managed VMs or Kubernetes — We chose Cloud Run over GKE specifically for cost efficiency. There is no cluster management overhead, no idle node cost, no master node fees. Cloud Run auto-scales to zero.
  • Single Region — Running in one region (Montreal) minimizes data egress costs and avoids the 2-3x cost multiplier of multi-region Firestore.
  • Document TTL — Firestore documents expire after 30 days (application-managed expireOn field), preventing unbounded storage growth.
  • SIMULATED Mode — The Enrichment Service supports pharmanet.endpoint=SIMULATED for development and testing, avoiding costs and dependencies on the real PharmaNet endpoint.

Where the Gaps Are

  • Polling Reads (Firestore Cost Multiplier) — Each transaction generates ~226 Firestore reads (polling loop). At $0.06/100K reads, this is ~$0.000136 per transaction. At 10,000 transactions/day, that is $1.36/day in polling reads alone. Switching to real-time listeners would reduce this by ~99%.
  • Always-On Enrichment Service — min-instances=1 means the Enrichment Service always has one container running, even during zero-traffic periods. This costs approximately $0.05/hour or ~$36/month for an idle container.
  • No GCP Budget Alerts — No budget has been configured in GCP Billing. There is no alert if costs exceed expected thresholds.
  • No Firestore Native TTL — The expireOn field exists but Firestore’s native TTL policy is not enabled. Without it, expired documents remain until a manual cleanup process runs.
  • No Cost Attribution — No resource labeling for cost allocation per service, per environment, or per message type.

How to Fill the Gap

  1. Replace Polling with Firestore Listener — Reduces Firestore read costs by ~99%. The single biggest cost optimization available.
  2. Enable Firestore Native TTL — Run: gcloud firestore fields ttls update expireOn --collection-group=dis-bc-responses --enable-ttl. Firestore automatically deletes expired documents, eliminating the need for a cleanup process and reducing storage costs.
  3. GCP Budget Alerts — Set up in Cloud Billing:
    • Alert at 50%, 80%, and 100% of monthly budget
    • Notifications to email and Pub/Sub for automated response
  4. Resource Labels — Add labels to Cloud Run services (service=dataservice, env=prod), Firestore, and Pub/Sub resources for cost breakdown in billing reports.
  5. Evaluate min-instances=0 — For non-production environments, set min-instances=0 to eliminate idle costs. Accept cold-start latency (~2-5s) in dev/staging.
Estimated: 1 day for budget alerts + Firestore TTL; 2 days for polling replacement

Q9: How Do You Make NFRs Real Instead of Theoretical?

9
Making Non-Functional Requirements Real
Partially Implemented

What We Built

Several NFRs are enforced in code today, not just documented:

NFR CategoryImplementationWhere in Code
Reliability Pub/Sub message staleness check — rejects messages older than 120 seconds PharmaNetMessageController.handlePubSubPush() checks publishTime + timeout.seconds
Data Retention Firestore documents expire after 30 days via application-managed TTL FirestoreService.saveResponse() sets expireOn = now + db.expire-in-days
Latency SLA Transaction timeout: 45 seconds (225 retries x 200ms polling interval) BCPublishController.fetchResponse() and FirestoreBcResponseFetcher
Latency SLA WebClient HTTP timeout: 60 seconds for outbound calls WebClientConfig.java configures responseTimeout(Duration.ofSeconds(60))
Idempotency Firestore document ID = messageId. Duplicate processing overwrites, not duplicates. FirestoreService.saveResponse() uses document(messageId).set()
Unique IDs Cryptographically random message IDs via SecureRandom BCPayloadBuilder.generateRandomId()
API Documentation SpringDoc OpenAPI / Swagger UI on Data + Enrichment services springdoc-openapi-starter-webmvc-ui in pom.xml
Testability 46 unit tests + contract tests in Data Service; SIMULATED mode in Enrichment Data Service src/test/; Enrichment pharmanet.endpoint=SIMULATED

Where the Gaps Are

  • No Spring Boot Actuator — None of the services include spring-boot-starter-actuator. There are no /health, /info, /metrics, or /prometheus endpoints. Cloud Run has no way to probe application health beyond “did the HTTP request succeed.”
  • No Circuit Breakers — No Resilience4j, Spring Retry, or Hystrix. A Firestore outage cascades to every in-flight transaction without graceful degradation.
  • No Distributed Tracing — No OpenTelemetry, Cloud Trace, or Zipkin. There is no way to trace a single transaction across Client → Data Service → Pub/Sub → Enrichment Service → PharmaNet → Firestore. Debugging latency issues requires manual log correlation.
  • No Prometheus / Micrometer Metrics — No custom application metrics (e.g., transactions per message type, PharmaNet response time, builder execution time).
  • No Docker HEALTHCHECK — Both Dockerfiles are minimal (FROM temurin:23-jre, COPY app.jar, java -jar). No HEALTHCHECK instruction.
  • No SLA Monitoring or Alerting — The 45-second transaction SLA and 99.9% availability target exist only in documentation. There is no dashboard or alert that fires when these are breached.
  • NFRs Documented but Not Enforced — Security (authentication, authorization, PHI protection) is defined in the hardening plan but not yet implemented in code. This makes them theoretical, not real.

How to Fill the Gap

Principle: An NFR is not real until it is (a) implemented in code, (b) monitored by a metric, and (c) enforced by an alert.

  1. Spring Boot Actuator — Add spring-boot-starter-actuator to both services. Expose /actuator/health, /actuator/info, /actuator/metrics. Configure Cloud Run startup and liveness probes: 
    gcloud run deploy ... \
  --startup-probe-path=/actuator/health \
  --liveness-probe-path=/actuator/health
  2. Micrometer + Cloud Monitoring — Add micrometer-registry-stackdriver to export custom metrics (transactions per event type, PharmaNet call duration, Firestore latency) to Cloud Monitoring.
  3. Cloud Trace — Add spring-cloud-gcp-starter-trace for automatic span propagation. A single transaction ID will be visible across all three services in the Cloud Trace UI.
  4. Resilience4j Circuit Breaker — Add resilience4j-spring-boot3 to both services. Configure circuit breakers for:
    • Firestore read/write operations (open after 50% failure over 10 calls)
    • PharmaNet HTTP calls (open after 3 consecutive failures)
  5. SLA Dashboard — Create a Cloud Monitoring dashboard with:
    • P50/P95/P99 transaction latency (target: P95 < 10s)
    • Error rate (target: < 0.1%)
    • Availability (target: 99.9%)
    Configure alerts when any SLA metric breaches its threshold for 5 consecutive minutes.
Estimated: 2 days (Phase 2.3 + resilience additions)
