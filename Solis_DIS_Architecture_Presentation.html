1. Executive Summary

The Solis Drug Information System (DIS) is a cloud-native, microservices-based platform designed to facilitate the integration between pharmacy software systems and British Columbia's PharmaNet network. The system handles two primary domains:

  • Adjudication (CPhA) — Processing pharmacy claims using the Canadian Pharmacists Association (CPhA) standard format for drug adjudication and reconciliation.
  • eHealth / BC PharmaNet — Exchanging HL7 v2 and HL7 v3 messages with PharmaNet for patient demographics, medication profiles, prescriptions, dispense events, and other mandatory pharmacy transactions.

The architecture is built entirely on Google Cloud Platform (GCP), leveraging Cloud Run for serverless compute, Pub/Sub for asynchronous messaging, and Firestore for persistent storage. The design ensures the system is stateless, asynchronous, and fault-resilient, capable of handling high-throughput pharmacy transactions with reliable delivery guarantees.

2. System Architecture Overview

The Solis DIS platform consists of three microservices working together in a decoupled, event-driven architecture:

Architecture Flow:

[Client Service] → HTTP POST → [Data Service] → Pub/Sub Publish → [Enrichment Service]

[Enrichment Service] → Builds HL7 Message → Calls PharmaNet/PCR → Saves Response to Firestore

[Data Service] → Polls Firestore for response → Returns to [Client Service]

High-Level Flow

  1. The Client Service (UI layer) sends a JSON request via HTTP to the Data Service.
  2. The Data Service publishes the message to a Google Cloud Pub/Sub topic and begins polling Firestore for a response.
  3. The Enrichment Service receives the message via Pub/Sub push subscription, enriches the payload, builds the appropriate HL7 v2 or HL7 v3 message, and sends it to PharmaNet or the Provincial Client Registry (PCR).
  4. The Enrichment Service writes the complete response (including the built HL7 request, the PharmaNet/PCR response, and metadata) to Firestore.
  5. The Data Service detects the response in Firestore and returns the full document back to the Client Service.
  6. The Client Service renders the response in the UI, showing the HL7 Request, HL7 Response, and the full database document.

AttributeDetails

4b. Data Service (solis-dis-dataservice)

AttributeDetails

RoleAPI gateway / Message orchestrator
Cloud Run URLsolis-dis-dataservice-809287589165.northamerica-northeast1.run.app

Responsibilities:

  • Receives incoming JSON requests from the Client Service at /api/dis/bc/publish.

  • Publishes the sanitized message to a Google Cloud Pub/Sub topic.
  • Polls Firestore for the response document using the message ID as the lookup key.

4c. Enrichment Service (solis-dis-enrichmentservice)

AttributeDetails
Port8080
RoleMessage processor / HL7 builder / PharmaNet gateway
Cloud Run URLsolis-dis-enrichmentservice (Cloud Run, push subscription target)

Responsibilities:

  • Receives messages from Pub/Sub via push subscription (HTTP POST from Pub/Sub to the Cloud Run endpoint).
  • Routes messages to the appropriate HL7 builder based on Event.eventTypeCode:
    • HL7 v3 messages (Find Candidates, Get Demographics) → builds SOAP/XML HL7 v3 messages for the Provincial Client Registry (PCR).
    • HL7 v2 messages (TRP, TIP, TRX, TDU/TAC, TPI, TIL, DISPENSE) → builds pipe-delimited HL7 v2 segments (MSH, ZCA, ZCB, ZCC, ZPX, ZCD, ZZZ, etc.) for PharmaNet.
  • Sends the constructed HL7 message to PharmaNet or the PCR.
  • Saves the complete result to Firestore, including:
    • hl7Request — The HL7 message that was built and sent.
    • hl7Response — The raw response received from PharmaNet/PCR.
    • Status, timestamps, and error details (if any).
  • Handles JSON escape sequence issues (e.g., \E in HL7 encoding characters) via a character-by-character parser in StringUtils.escapeInvalidBackslashes().

5. Google Cloud Pub/Sub — Message Broker

Google Cloud Pub/Sub serves as the central asynchronous message broker between the Data Service and the Enrichment Service. It decouples the two services completely, enabling independent scaling and fault isolation.

How Pub/Sub Is Used

AspectDetails
Publisher Data Service publishes JSON messages to a Pub/Sub topic upon receiving a request from the Client Service.
Subscriber Enrichment Service subscribes via a push subscription. Pub/Sub delivers messages as HTTP POST requests to the Enrichment Service's Cloud Run endpoint.
Delivery Mode Push (changed from pull for cost optimization). Pub/Sub pushes messages to the Enrichment Service, eliminating the need for the service to continuously poll.
Message Format JSON payload containing Header, Sender, Receiver, Candidate, Event, MSH, ZCA, ZCB, ZCC, ZPX, ZCD, ZZZ segments, and other PharmaNet-specific fields.
Message ID Each message carries a unique messageId (20-character random string) used as the correlation key for Firestore lookups.
Retry / Dead-Letter Pub/Sub automatically retries failed deliveries with exponential backoff. Messages that cannot be processed are retained per the subscription's acknowledgment deadline.

Benefits of Push Over Pull

  • Cost savings: No idle polling threads consuming Cloud Run CPU. The Enrichment Service only activates when a message arrives.
  • Lower latency: Messages are delivered immediately upon publish, rather than waiting for the next poll cycle.
  • Scales to zero: Cloud Run instances can scale to zero when there are no messages, reducing costs to near-zero during idle periods.

6. Google Cloud Firestore — Database

Google Cloud Firestore (NoSQL document database) is used as the shared state store between the Data Service and the Enrichment Service. It enables the asynchronous request-response correlation pattern.

How Firestore Is Used

OperationServiceDetails
Write Enrichment Service After processing the HL7 message and receiving a response from PharmaNet/PCR, the Enrichment Service writes the complete result document to Firestore, keyed by messageId.
Read (Poll) Data Service After publishing to Pub/Sub, the Data Service polls Firestore repeatedly (with configurable timeout, default 120s) looking for a document with the matching messageId.
Read (Return) Data Service → Client Once found, the full Firestore document is returned as JSON to the Client Service, which displays it in the UI.

Firestore Document Structure

Collection: [messageType]
Document ID: [messageId]

Fields:
  hl7Request — The HL7 v2/v3 message built by the Enrichment Service
  hl7Response — The raw response from PharmaNet or PCR
  status — Processing status (success, error)
  timestamp — When the response was written
  originalRequest — The original JSON request payload
  error — Error details (if processing failed)

7. Asynchronous, Stateless & Fault-Resilient Design

7.1 Asynchronous Processing

The system is fundamentally asynchronous. No service blocks on another service's processing:

  • The Data Service publishes a message to Pub/Sub and immediately begins polling Firestore — it does not wait for the Enrichment Service to process the message synchronously.
  • The Enrichment Service processes messages independently, triggered by Pub/Sub push delivery. It writes results to Firestore and acknowledges the Pub/Sub message.
  • The correlation between request and response happens through Firestore using the unique messageId, not through direct service-to-service calls.

7.2 Stateless Services

All three services are completely stateless:

  • No in-memory session state is stored between requests.
  • Each request carries all necessary context (message ID, event type, patient data) in the JSON payload.
  • Services can be restarted, scaled up, or scaled down without losing any data.
  • Cloud Run enforces statelessness — instances are ephemeral and may be terminated at any time.
  • All persistent state lives in Firestore (response data) and Pub/Sub (in-flight messages).

7.3 Fault Resilience

Failure ScenarioHow It's Handled
Enrichment Service crashes mid-processing Pub/Sub does not receive an ACK, so it redelivers the message to another instance (or the same instance after restart). No data is lost.
Data Service times out waiting The Data Service returns a timeout error to the Client. The user can click Retry to re-poll Firestore — if the Enrichment Service completed in the meantime, the response is available.
Firestore write failure The Enrichment Service logs the error and does not ACK the Pub/Sub message, triggering redelivery and a retry of the entire processing.
Network failure to PharmaNet The Enrichment Service writes an error document to Firestore with the failure details, allowing the client to see the exact error and retry.
Cloud Run cold start delay Mitigated by setting --min-instances=1 on the Enrichment Service to keep at least one warm instance.
Invalid JSON / escape sequences Both Data Service (JsonSanitizer) and Enrichment Service (StringUtils) include character-by-character parsers that fix invalid escape sequences (e.g., \E in HL7 encoding characters).
WebClient HTTP errors All HTTP calls use try-catch with structured error responses returned as JSON, preventing cascading failures.
Key Design Principle: The combination of Pub/Sub's at-least-once delivery guarantee, Firestore's durable storage, and stateless Cloud Run services ensures that no message is ever lost, even during partial outages, cold starts, or service restarts.

8. Message Flow — End-to-End Request Lifecycle

Below is the step-by-step lifecycle of a PharmaNet HL7 message (e.g., a Patient Profile request):

StepComponentAction
1User (Browser)User selects "Patient Profile (TRP)" from the landing page and clicks Submit JSON.
2Client ServiceSends HTTP POST to Data Service at /api/dis/bc/publish with the JSON payload containing Header, MSH, ZCA, ZCB, ZCC, ZPX, ZCD, Event, and other segments.
3Data ServiceReceives the request, sanitizes the JSON, and publishes the message to the Pub/Sub topic with the messageId as an attribute.
4Pub/SubDelivers the message via push subscription (HTTP POST) to the Enrichment Service's Cloud Run endpoint.
5Enrichment ServiceReceives the message, determines the event type (TRP), and routes to the HL7 v2 builder.
6Enrichment ServiceBuilds the complete HL7 v2 message (MSH|^~\&|...|ZCA|...|ZCB|...) from the JSON fields.
7Enrichment ServiceSends the HL7 v2 message to PharmaNet and receives the response.
8Enrichment ServiceWrites the complete result to Firestore: { hl7Request, hl7Response, status, timestamp, ... }.
9Enrichment ServiceAcknowledges (ACKs) the Pub/Sub message.
10Data ServiceDetects the new Firestore document via polling and returns the full document JSON to the Client Service.
11Client ServiceParses the response, extracts hl7Request and hl7Response, and renders all three panels: HL7 Request (blue), HL7 Response (green), Full Response from Database (orange).
